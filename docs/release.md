# Releasing DuoBudget

DuoBudget ships as a Flutter app for **Android** and **desktop** (Windows,
macOS, Linux). There are no servers or accounts to deploy — a release is just
signed, packaged binaries.

> **Where to look.** [`distribution.md`](distribution.md) is the authoritative
> guide to how DuoBudget is distributed (GitHub Releases only), the CI pipeline
> that builds every artifact on a tag, the pinned toolchain, and the
> telemetry-free download metrics. **This** page is the deeper reference for the
> one-time **upload-keystore creation** and the optional **Play Store `.aab`**
> path. Start with `distribution.md` unless you need those.

Before packaging anything, from `app/`:

```bash
../check.sh        # dart analyze + flutter test
../tool/e2e.sh     # multi-hub sync convergence end-to-end
```

Bump the version in `app/pubspec.yaml` (`version: <name>+<build>`) — `versionName`
and `versionCode` on Android and the desktop bundle versions all derive from it.

Regenerate the placeholder branding if you have replaced the artwork:

```bash
dart run tool/generate_branding.dart
```

## Android (signed release)

> **Never generate or commit a keystore in this repository.** A keystore is a
> private signing credential. It lives outside the repo and is referenced by an
> uncommitted `android/key.properties` (already in `.gitignore`, along with
> `*.jks` / `*.keystore`). Losing it means you can never update the app on Play;
> leaking it lets anyone ship as you. Back it up somewhere safe and private.

### 1. Create an upload keystore (one time, on your machine — not in CI)

```bash
keytool -genkey -v \
  -keystore ~/duobudget-upload.jks \
  -storetype JKS \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -alias upload
```

`keytool` ships with the JDK. It will prompt for a store password, a key
password, and your details. Store the resulting `.jks` file and both passwords
in a password manager or secret store.

### 2. Point the build at it

Create `app/android/key.properties` (uncommitted):

```properties
storeFile=/absolute/path/to/duobudget-upload.jks
storePassword=<store password>
keyAlias=upload
keyPassword=<key password>
```

`android/app/build.gradle.kts` reads this file when present and signs the release
build with it; when it is absent the build falls back to debug signing so
`flutter run --release` still works locally. Nothing secret is ever read from the
repo.

In CI, write `key.properties` and the `.jks` from your secret store at build
time and delete them afterward — do not check them in.

### 3. Build

App bundle for the Play Store:

```bash
cd app
flutter build appbundle --release      # build/app/outputs/bundle/release/app-release.aab
```

Or a standalone APK (for sideloading / non-Play distribution):

```bash
flutter build apk --release            # build/app/outputs/flutter-apk/app-release.apk
```

### 4. Verify the signature

```bash
jarsigner -verify -verbose -certs \
  build/app/outputs/flutter-apk/app-release.apk
```

Then upload the `.aab` to the Play Console (or distribute the `.apk` directly).

## Desktop

Desktop builds are unsigned by default here; platform code-signing/notarization
is optional and noted per platform. Each platform must be built **on that
platform** (Flutter does not cross-compile desktop binaries).

### Windows

Requires Visual Studio with the "Desktop development with C++" workload.

```bash
cd app
flutter build windows --release
```

Output: `build/windows/x64/runner/Release/` (the `.exe` plus its DLLs and the
`data/` folder — ship the whole directory).

- **Package:** zip the `Release/` folder, or wrap it with an installer
  (Inno Setup or MSIX). For MSIX, add the `msix` dev-dependency and run
  `dart run msix:create`.
- **Sign (optional):** `signtool sign /fd SHA256 /a <artifact>` with an
  Authenticode certificate.
- The app icon comes from `windows/runner/resources/app_icon.ico`.

### macOS

Requires Xcode.

```bash
cd app
flutter build macos --release
```

Output: `build/macos/Build/Products/Release/duobudget.app` (the bundle's
`PRODUCT_NAME` is lowercase `duobudget`).

- **Sign & notarize (optional but needed for distribution outside your own
  machine):**

  ```bash
  codesign --deep --force --options runtime \
    --sign "Developer ID Application: <Your Name> (<TEAMID>)" \
    build/macos/Build/Products/Release/duobudget.app

  # zip it, then submit for notarization:
  xcrun notarytool submit duobudget.zip \
    --apple-id <you@example.com> --team-id <TEAMID> --wait
  xcrun stapler staple build/macos/Build/Products/Release/duobudget.app
  ```

- **Package:** create a `.dmg` (e.g. with `create-dmg`) or a zip.
- The app icon comes from
  `macos/Runner/Assets.xcassets/AppIcon.appiconset/` (regenerated by
  `tool/generate_branding.dart`).

### Linux

Requires `clang`, `cmake`, `ninja-build`, `pkg-config`, `libgtk-3-dev` (plus
`libsqlite3-dev` for the drift database at runtime).

```bash
cd app
flutter build linux --release
```

Output: `build/linux/x64/release/bundle/` (the executable plus `lib/` and
`data/` — ship the whole directory).

- **Package:** tar the `bundle/` directory, or build a Flatpak / Snap / AppImage
  from it for wider distribution.
- Ensure the target machine has `libsqlite3` available (bundled by most distros).

## Post-build smoke test

On each platform, launch the packaged build and confirm:

1. First-run setup completes and the dashboard shows the new-household empty
   state.
2. A desktop can **start a hub** (Manage → Sync & hubs) and a second device can
   **pair** and sync.
3. Export a `.dbevents.zip` and re-import it on a fresh install — the state
   matches.
